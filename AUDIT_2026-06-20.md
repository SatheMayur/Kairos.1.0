# Recruitment CRM — Production Audit
**Date:** 2026-06-20 · **System:** kgirdharlal-recruitment.vercel.app (FastAPI + SQLAlchemy async, Neon Postgres, Jinja/vanilla-JS UI, Baileys WhatsApp bridge, Gemini)
**Method:** Live API inspection + source-code review of the actual models/endpoints. Evidence is separated into **Confirmed Fact** (verified in code or live data), **Possible Issue** (architecturally implied, not directly observed), and **Assumption (verify)**.

---

## 1. Executive Summary

The CRM is in materially better shape than the issue list assumes — several of the listed concerns (duplicate applications, "interview scheduled before confirmation", no-contact candidates in active lists) were **already fixed** in prior work and I verified the fixes in code. The audit confirms those, then identifies the **genuinely open** problems.

**Top open risks (verified):**

1. **CRITICAL — Job delete will 500 / orphan data.** `DELETE /jobs/{id}` deletes the job row with **no cleanup of dependent rows** (`shortlist`, `outreach_log`, `interviews` all FK to `jobs.id`). On Postgres this throws a foreign-key violation. Candidate delete *does* cascade; job delete does **not**. This is the "wrong jobs cannot be removed" symptom (Issue 5).
2. **HIGH — Outreach ignores job status and review state.** The outreach engine contacts every reachable `SHORTLISTED` **and** `PENDING` entry **regardless of whether the job is PAUSED**. Live data: 19 reachable candidates sit under **paused** jobs (Graphic Designer, Electronic Engineer) and would be messaged about roles that are on hold; 28 `PENDING` (un-reviewed) HR candidates would be auto-contacted before a human shortlists them (Issues 6, 9-adjacent).
3. **HIGH — Communication log has no `direction` and no `status_history`.** Inbound replies are stored as `reply_text` on the *outbound* row, not as first-class inbound messages; status changes are not audited at all. This weakens the "auditable outreach" and "every status backed by real data" goals (Issues 1, 3, 10).
4. **MEDIUM — Dashboard "Shortlisted" is cumulative and includes DROPPED.** The count mixes "currently shortlisted" with "ever reached shortlist," so a card labelled *Shortlisted* can exceed the number of people actually in that stage (Issue 8).
5. **MEDIUM — "Stuck PENDING" pool.** Job 3 has **16 PENDING but 0 reachable** — locked Apna profiles that can never advance and are never auto-closed; they inflate "candidates for this job" without being actionable (Issues 2, 6).

**Already fixed (verified, no action needed):** duplicate candidate-job rows (unique constraint + upsert + startup dedupe); interview "scheduled" only with a backing dated record; no-contact candidates hidden from active lists; job-centric discovery page exists.

## 5. Recruitment Workflow Issues

- **WF-1 (High):** Contacting paused-job candidates and un-reviewed PENDING (BL-1/BL-2). A "Contact all" or the daily cron currently does both. *Recommended:* outreach targets only **SHORTLISTED on ACTIVE jobs** by default; PENDING and paused jobs require an explicit opt-in.
- **WF-2 (Medium):** No "review queue → approve → outreach" gate distinct from scoring. The Discovery page's *Shortlist* action is the right primitive; make it the **only** path into auto-outreach.
- **WF-3 (Medium):** Re-sourcing re-scores everyone; combined with auto-outreach-on-source this could re-message if a status were ever reset. Mitigated by F2 + idempotent CONTACTED check, but worth a regression test.
- **WF-4 (Low):** Job 3 hired=1 but 16 PENDING remain open — no "close remaining candidates when a role is filled" prompt.

## 7. Candidate Data Quality Audit

- **Confirmed:** `is_reachable` gate exists; no-contact candidates are hidden from Candidates list, Shortlist, Pipeline, Outreach, and shown in **Needs Fixing** with plain-English fixes (incl. Apna "unlock" guidance). Outreach refuses unreachable candidates (`/candidates/{id}/contact` → 422).
- **Open:** unreachable candidates are still **counted** in per-job totals (DI-5); there's no lifecycle that auto-parks them or expires them.
- **Rules to formalize:**
  1. *Outreach-ready* = `is_reachable` **and** status ∈ {SHORTLISTED} **and** job ACTIVE.
  2. *Interview-schedulable* = has confirmed slot **and** reachable.
  3. *Park* (don't delete): unreachable > N days with no path to contact → sub-state `NEEDS_CONTACT_INFO`, excluded from actionable counts.
  4. *Delete* only on explicit recruiter action with a dependency check (never bulk-auto).

## 8. Conversation Audit (Issue 1)

**Root cause of "same candidate appears multiple times":** the Outreach log is **message/event-level by design** (INITIAL_CONTACT, FOLLOW_UP×2, REMINDER each = a row; F3), while a true *conversation* lives in the separate `Conversation` table (F5). The recruiter sees the raw event log, not a collapsed thread. The WhatsApp dashboard already collapses (`_collapse_threads`), but the **Outreach page does not**.

**Correct model:**
- **One conversation per (candidate, job).** Status changes do **not** create rows.
- **Messages** are children of the conversation, each with `direction` (IN/OUT), `channel`, `status`, `at`.
- **Recruiter inbox** = list of conversations (one row each), sorted by last activity, showing last message + unread/needs-human flag; opening it shows the message timeline.

## 9. Interview Workflow Audit (Issue 9)

**Confirmed correct already (F8):** a proposed slot does **not** mark the candidate INTERVIEW_SCHEDULED; they stay INTERESTED until a slot is confirmed (`scheduled_at` set), and orphaned "scheduled" rows are reconciled down on startup.

**Recommended lifecycle (formalize the existing behavior):**
`PROPOSED → AWAITING_CONFIRMATION → CONFIRMED → COMPLETED → FEEDBACK_SUBMITTED` (+ `CANCELLED`, `NO_SHOW`).

**Counting rules:**
- *Today's / Upcoming Interviews, Reminders, Dashboards:* count **CONFIRMED only** (`scheduled_at NOT NULL`).
- *Awaiting Confirmation:* its **own** section ("Slots sent, waiting on candidate"), never in Today/Upcoming.
- **Verify:** the Interviews dashboard and any "Interviews" KPI filter on `scheduled_at IS NOT NULL` (Issue 8 / Metric "Interviews").

## 10. Job Management Audit (Issue 5)

- **DI-1 is the headline:** job delete is unsafe (F9).
- **No archive:** CLOSED exists but closed jobs aren't visibly separated, and delete is the only "removal", which is the unsafe path.
- **Recommended:**
  - **Archive (default):** set `CLOSED` + `archived_at`; hide from active lists; keep all history. Reversible.
  - **Delete (guarded):** allowed only when the job has **zero applications**; otherwise force archive. When permitted, cascade dependents in one transaction (mirror `delete_candidate`).
  - **Pause semantics:** PAUSED must **suspend outreach** (fixes BL-1).

## 11. Database Recommendations

Current → recommended mapping (pragmatic; current `shortlist` already does the job of two tables well):

```
jobs                (exists)  + archived_at, ON DELETE handling
candidates          (exists)  + reachable (computed), needs_contact_info flag
candidate_applications        = current `shortlist` (rename conceptually).
                                One row per (candidate_id, job_id). UNIQUE already enforced (F1).
                                Holds pipeline status. THIS is "1 application", not many.
candidate_matches             = score data. Today it's embedded in shortlist.score/score_breakdown.
                                Keep embedded unless you need score history; if you do, split:
                                candidate_matches(application_id, score, breakdown, model, scored_at).
conversations       (exists, extend) (candidate_id, job_id) UNIQUE, status, needs_human, last_msg_at
messages            (NEW)     (conversation_id, direction IN/OUT, channel, body, status,
                                provider_id, error_detail, created_at, delivered_at, read_at)
                                — replaces overloading outreach_log.reply_text (DI-4).
communications      = view over messages (channel+status+timestamp+direction+outcome) for audit.
interviews          (exists, extend) status per §9; scheduled_at NULL until confirmed.
interview_slots     (NEW)     (interview_id, proposed_for, chosen bool) — model multi-slot proposals.
status_history      (NEW)     (application_id, from_status, to_status, actor, reason, created_at) (DI-3)
```

**Relationships:** `jobs 1—* candidate_applications *—1 candidates`; `candidate_applications 1—1 conversations`; `conversations 1—* messages`; `candidate_applications 1—* interviews 1—* interview_slots`; `candidate_applications 1—* status_history`. Keep `outreach_log` as the send-attempt ledger or migrate it into `messages` (recommended end-state).

## 12. API Recommendations

Already present (verified): `GET /jobs`, `GET /jobs/{id}/stats`, `GET /jobs/{id}/candidates`, `GET /shortlist`, `GET /candidates/{id}/profile`, `GET /candidates/{id}/conversation`, `POST /candidates/{id}/contact`, `POST /jobs/{id}/contact-all`, cron endpoints.

Recommended additions/changes:
- `POST /jobs` from structured fields (Company/Title/Location/Experience) → **AI-generate JD** → review → create (Issue 4). New: `POST /jobs/generate-jd` (returns draft JD + parsed fields, does not persist).
- `DELETE /jobs/{id}` → **guarded**: 409 if applications exist (suggest archive); `POST /jobs/{id}/archive`, `POST /jobs/{id}/reopen`.
- `GET /conversations?job_id=&needs_human=` (thread list = inbox), `GET /conversations/{id}/messages`, `POST /conversations/{id}/reply`.
- `PATCH /applications/{id}/status` writes `status_history` (single source of truth for stage moves).
- Outreach scoped: `POST /jobs/{id}/contact-all?status=SHORTLISTED&include_paused=false` (defaults that fix BL-1/BL-2).
- `GET /metrics/dashboard` returning each metric tagged `{value, basis: current|cumulative, unit: people|pairs}` (Issue 8).

## 16. Final Recommendations

1. **Fix job delete first** — it's the only Critical, and it's the literal "wrong jobs can't be removed" symptom.
2. **Make PAUSED mean paused** and **make SHORTLISTED the gate for auto-outreach** — these two changes prevent the most likely *wrong outward action* (messaging the wrong people). This is exactly why I held the staged send today.
3. **Label every metric's basis** — the data is mostly correct; the *meaning* is what misleads.
4. **Promote conversations + add `direction`/`status_history`** to make outreach and stage changes truly auditable.
5. Treat **delete as the exception, archive as the default** everywhere (jobs and candidates), with dependency checks — no bulk auto-deletes.

**Net:** the engine and data model are sound and the previously-reported duplication/interview-state bugs are genuinely fixed. The remaining work is about **guardrails (delete, paused, review-gate)** and **honest reporting (metric basis, audit trail)** — not a rebuild.
